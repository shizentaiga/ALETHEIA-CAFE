# エリア検索・現在地連携：統合実装計画書 (v1.8)

## 1. 設計思想：ステートレス & 直感的動線
- **拡張性のあるパッキング**: `area=auto:13123` 形式を採用。将来の属性拡張（駅・カテゴリ等）を阻害しない。
- **動線上の「現在地」**: ユーザーの視線（エリア選択 → キーワード入力）の間に「⊕」を配置し、思考を妨げず利便性を向上。
- **堅牢なバリデーション**: 不正なURL（NaN等）を排除するセーフティなパース処理。

---

## 2. パラメータ設計とパース論理

| パラメータ形式 | 意味 | 内部ステータス |
| :--- | :--- | :--- |
| `?area=13123` | 手動選択 | `{ areaId: 13123, isAuto: false }` |
| `?area=auto:13123` | 現在地由来 | `{ areaId: 13123, isAuto: true }` |

**パース時の安全策:**
- `split(':')` 後の ID を `parseInt`。
- `Number.isFinite` でチェックし、不正値（NaN/Infinity）は一律 `null` へフォールバック。

---

## 3. 実装の要諦

### ① `lib/searchUtils.ts` (Core Logic)
- **距離計算**: 三平方の定理を用い、日本付近の緯度・経度比率（0.8）を考慮した近似計算。
- **スケーリング対応**: データ量増大時は、計算処理をサーバーサイド（D1）の空間関数へオフロード可能な設計。

### ② `db/migration.ts` (Data Integrity)
- **正規化バッチ**: `@geolonia/normalize-japanese-addresses` を採用。
- **フロー**: `address` 取得 → 正規化(pref/city) → `areas` マスタ照合 → `area_id` 付与。

---

## 4. 現在地取得（Geolocation）のUX設計

### デザイン・配置
- **位置**: エリア選択ドリルダウンとキーワード入力窓の間に「⊕」ボタンを設置。
- **スタイル**: 背景透明・サブ機能として馴染むサイズ感。スマホ時は誤タップ防止のため適度な `gap` を確保。

### 取得シーケンス
1. **オンデマンド実行**: ユーザーのクリック時のみ `navigator.geolocation` を呼び出す（不快感・権限要求の最小化）。
2. **フィードバック**: 取得中はアイコンを反転・回転させ、フリーズを防止。
3. **マッピング**: 取得した座標から「最短距離の `area_id`」を特定し、URLを `?area=auto:{id}` に更新。

## 5. ロードマップ

### Phase 1: 高精度マスタと検索基盤
- 住所データの正規化と `areas` テーブルの構築。
- 文字列検索から `area_id` 検索への完全移行。

### Phase 2: Geolocation 連携
- 現在地取得ボタンの配置と、クライアントサイドでの最短エリア特定ロジックの実装。
- 取得後のラベル表示（「📍 横浜市付近」等）の動的更新。

### Phase 3: パフォーマンス最適化
- `areas` リストの軽量化。
- モバイル・PWA 向けの UI/UX 微調整（案A：ドリルダウン内への統合検討）。

---

## 6. 懸念事項と対策

- **バリデーション**: `NaN` 対策を施したパースロジックにより、不正パラメータによるシステム停止を防止。
- **精度**: 正規化ライブラリの変換結果を目視確認し、住所パース漏れを最小化。
- **負荷**: 初期は軽量JSONで対応。将来的な数千件規模のエリア拡大時は `GET /api/nearest-area` へ移行。
