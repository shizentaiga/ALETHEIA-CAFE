# エリア検索・現在地連携：統合実装計画書 (v1.9)

## 1. 設計思想：オフライン・ファースト & ゼロ・レイテンシ
- **脱・外部API依存**: 座標から住所IDの特定を外部API（HeartRails等）に頼らず、ローカルDB（D1）内の座標データを用いて完結。
- **代表点（Centroid）マッチング**: 市区町村の中央値座標を用いた近傍探索により、ミリ秒単位の高速レスポンスを実現。
- **ステートレスな導線**: area=auto:{id} 形式を維持し、URLに「現在地由来」のコンテキストを保持。

---

## 2. 座標データ構造（D1 / areas テーブル）
既存の areas テーブルに座標カラムを追加し、空間検索の基盤とする。

| カラム名 | 型 | 用途 |
| :--- | :--- | :--- |
| id | INTEGER | プライマリキー |
| name | TEXT | 市区町村名（例：江戸川区） |
| lat | REAL | 市区町村の地理的中央値（緯度） |
| lng | REAL | 市区町村の地理的中央値（経度） |

---

## 3. 実装の要諦：現在地判定ロジック

### ① 距離計算アルゴリズム
ブラウザで取得した座標を、以下の優先順位でエリアIDに変換する。
1. **初期実装（Client）**: areas のリスト（ID/座標のペア）をフロントエンドで保持し、三平方の定理の近似値で最短距離の ID を特定。
2. **計算式**: dist = (dy * dy) + (dx * 0.8)^2  ※日本付近の経度補正。

### ② セーフティ・パース（バリデーション）
URLパラメータから取得したIDが不正な場合のフォールバックを徹底。
- idStr を parseInt し、Number.isFinite で判定。
- NaN や不正な文字列の場合は、サイレントに null を返し、エラー落ちを防止。

---

## 4. UI/UX：動線とフィードバック

### ⊕ ボタン（現在地取得）の振る舞い
1. **配置**: エリアドリルダウンの右側に配置。キーワード入力窓への視線動線上に設置。
2. **取得中状態**: アイコンをスピン（回転）させ、非同期処理中であることを明示。
3. **結果反映**:
   - 最短エリアID（例：13123）を特定し、URLを /?area=auto:13123 へ更新。
   - HTMXによる部分スワップで検索結果を即時表示。
   - ドリルダウンのラベルを「📍 現在地（江戸川区）」へ動的更新。

---

## 5. ロードマップ（v1.9 改訂版）

### Phase 1: 座標付きエリアマスタの構築
- areas テーブルに主要な市区町村の代表点座標をインポート。
- 既存店舗データに area_id を付与する正規化バッチの実行。

### Phase 2: ローカル近傍検索の実装
- navigator.geolocation から最短の area_id を割り出す JS ロジックの実装。
- 外部APIへのリクエストを最小化し、パフォーマンスを極限まで高める。

### Phase 3: UI統合とレスポンシブ対応
- キーワード検索履歴とのパラメータ競合（デグレ）を防止。
- モバイル表示時、指の太さを考慮した gap 設定と誤タップ防止。

---

## 6. 検討事項と対策まとめ

- **境界線のズレ**: 中央値判定のため、境界付近では隣接区が出る可能性があるが、「現在地付近」という表現でUX上の納得感を担保。
- **HeartRails Geo APIの活用**: 座標から詳細な「駅名」や「町名」を補足的に取得したい場合のみ、補助的にコールする構成を検討。
- **スケーラビリティ**: エリア数が膨大になった場合は、D1 の SQL（ORDER BY 距離）による検索へシームレスに移行。
