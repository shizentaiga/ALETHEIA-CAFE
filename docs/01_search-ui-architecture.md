# Search UI Architecture & State Synchronization

## ファイル名: search-ui-architecture.md

## 1. 概要
本プロジェクトでは、HTMXによるサーバーサイドレンダリング（SSR）を軸としつつ、モダンなSPAに近い動的な検索体験（キーワードのチップ化・同期）を実現するために、**「State-on-HTML」**パターンを採用しています。

## 2. ディレクトリ構成と役割
UIの状態管理に関連する主要ファイルは以下の通りです。

- **`src/components/SearchResult.tsx` (State Provider)**
  - サーバーから返却された「正規化済みクエリ」を隠しフィールド (`input type="hidden"`) としてDOM内に保持します。

## 3. 同期ロジックのシーケンス
キーワード検索からチップが表示されるまでの流れは以下の通りです。

1. **ユーザー入力**: ヘッダーの検索窓から「小岩 小岩」と入力して検索。
2. **サーバー処理**: サーバー側でクエリを「小岩」に正規化し、検索結果と共に正規化済み文字列を返却。
3. **HTMX 部分更新**: `#search-result-module` が書き換わり、その中に新しい `current-q-state` が配置される。
4. **イベント発火**: `htmx:afterSettle` イベントを JS が検知。
5. **DOM生成 (Chipify)**: 
   - ヘッダー内の既存チップを消去し、新しいチップを生成。
   - 各チップに「その単語を除外した再検索用URL」を `hx-get` として付与。
6. **HTMX再登録**: `htmx.process()` を実行し、動的に生成された「×」ボタン（チップ削除ボタン）をHTMXの動作対象として有効化。

## 4. 設計上の重要ポイント

### 4.2 ステートの単一光源化
UIの状態（現在の検索語句など）をJSのメモリ上ではなく、常に**「DOM（Hidden Input）」**に置くことで、HTMXによる部分更新が行われても状態の不整合が起きないように設計されています。

### 4.3 ユーザー体験の最適化
- **プレースホルダーの動的制御**: チップが存在する場合は検索窓のプレースホルダーを非表示にし、チップが全消去された時のみ再表示します。
- **入力欄の自動クリア**: 確定したキーワードはチップとして表示されるため、入力欄自体は常に空の状態にリセットされ、次の入力（絞り込み）を促します。
