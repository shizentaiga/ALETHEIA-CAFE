# HTMX 実装の技術要約：状態を DOM に預ける設計

---

## 1. 「バケツ（隠しフィールド）」戦略

HTMX は状態をメモリ（JS 変数）ではなく、**DOM（HTML）** に持たせるのが基本。

- **コツ**：検索結果などの「動的に書き換わるエリア」の最上位に、現在の検索条件（`area`、`category` 等）を `<input type="hidden">` として配置する
- **メリット**：サーバーから HTML が返ってくるたびに状態が自動更新され、常に最新の「選択状態」が DOM 上に刻印される

---

## 2. `hx-include` によるパラメータの相互補完

複数のコンポーネント（ヘッダーの検索窓、サイドバーのエリア選択など）が独立している場合、`hx-include` で繋ぐ。

- **コツ**：送信元（Trigger）は自分の値だけでなく、`hx-include="他の入力欄の ID"` を使って、画面上の他の状態を「道連れ」にする
- **結果**：キーワード検索時にエリア情報を失わず、エリア選択時にキーワード情報を失わない「パラメータの合流」が容易に実現できる

---

## 3. URL と DOM の同期（SSoT の維持）

`hx-push-url="true"` または `HX-Push-Url` ヘッダーを活用し、ブラウザの URL を常に現在の検索状態と同期させる。

| タイミング | 処理 |
|:---|:---|
| 部分更新時 | HTMX が URL を書き換える |
| リロード時 | Hono が URL パラメータを見て、hidden フィールドを含めた初期 HTML を正しく生成する |

- **結果**：「戻る」ボタンやブックマークが正常に動作し、JS による履歴管理の手間が不要になる

---

## 4. 現場でのデバッグ・検証プロセス

HTMX の開発では JS の内部変数が見えないため、ブラウザの DevTools を活用した状態確認が必須。

### ① コンソールのペースト許可設定

ブラウザのセキュリティ保護により、初回はコードの貼り付けが拒否される。以下をコンソールに入力して Enter で解除する。

~~~text
allow pasting
~~~

### ② 状態確認デバッグコマンド

hidden input に意図した値が入っているかを JS で直接確認する。

~~~javascript
// 特定の ID の値を確認する（例：エリア情報）
document.querySelector('#current-area-state').value
~~~

**チェックポイント**：エリア選択などの操作直後にこのコマンドを叩き、値が `'東京都'` 等に書き換わっていれば DOM の状態管理は成功。

### ③ 要素のハイライト確認

~~~javascript
// 要素そのものを返す（マウスを乗せると画面上でハイライトされる）
document.querySelector('#current-area-state')
~~~

## 5. 通信と状態の循環フロー

~~~
① Action   ユーザーが操作（入力 or クリック）
      ↓
② Collect  hx-include が指定 ID から現在の値をかき集める
      ↓
③ Request  合流したパラメータでサーバーへリクエスト
      ↓
④ Response サーバーは新データ＋次の操作に備えた hidden フィールドを返す
      ↓
⑤ Update   DOM が書き換わり、新たな「状態（バケツ）」がセットされる
      ↓
    （① へ戻る）
~~~

---

## 6. 最小修正で済ませるためのポイント

- **疎結合**：コンポーネント間を Props で繋ぐのではなく、ID（セレクタ）で繋ぐ。親子関係に縛られない柔軟なレイアウト変更が可能になる
- **サーバーサイド駆動**：クライアント側で「URL を組み立てる JS」を書かない。パスの生成はサーバー側の属性定義（`hx-get`）に任せる

---

## 7. 技術的教訓

> 「状態をどこで管理するか？」という問いに対し、  
> React なら **State** と答えるところを、  
> HTMX では **「次にサーバーへ送るための HTML 要素」** と答えるのが、  
> コードを肥大化させない最大のコツである。

## 補足

### `hx-include` のセレクタ記法について

`hx-include` には ID 以外に CSS セレクタが使えるため、複数要素をまとめて巻き込む際は以下のように書ける。

~~~html
<!-- 名前属性でまとめて指定する例 -->
<input
  hx-get="/api/search"
  hx-include="[name='area_id'], [name='category_id'], [name='keyword']"
/>
~~~

hidden input が増えてくると ID 管理が煩雑になりやすい。`name` 属性ベースのセレクタで統一しておくと、後からフィールドを追加しても `hx-include` 側を修正せずに済む場面が多い。

### リロード対応（SSoT）の実装タイミングについて

§3 の「リロード時に Hono が URL パラメータを見て初期 HTML を生成する」処理は、実装コストが比較的高い。スタートアップ初期は `hx-push-url` を使わず URL 同期を後回しにし、コア機能が固まった段階で追加するのが現実的なトレードオフ。
