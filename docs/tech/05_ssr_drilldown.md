# 技術資料：ステートレス・ドリルダウン検索アーキテクチャ
〜SSR による堅牢性と HTMX による動的操作感の両立〜

---

## 1. 解決したい課題

従来のドリルダウン検索（階層構造の絞り込み）を SPA（Single Page Application）で実装する場合、階層の状態をメモリ内の State で管理することが一般的です。しかし、その手法には以下の課題がありました。

- **リロード耐性の欠如**：ページを更新すると選択状態がリセットされる
- **URL 共有の困難さ**：特定の階層を開いた状態を他人に共有できない
- **SSR との不整合**：初回描画時に UI の状態をサーバーと同期させるロジックが複雑になる

本設計では、**「URL を唯一の状態管理源（Single Source of Truth）」** と定義することで、これらの課題を解決し、高速かつ堅牢な検索体験を実現します。

---

## 2. アーキテクチャ全体像

システムは、URL に含まれるパラメータを「検索エンジンの条件」と「UI の表示状態」に分離して処理します。

~~~mermaid
graph TD
    URL["URL: ?area=10&parent_id=10"] --> |area=10| SSR["メインコンテンツ (SSR)"]
    URL --> |parent_id=10| HTMX["ドリルダウンUI (HTMX)"]
    subgraph Server_Response
        SSR --> |1. 高速描画| Browser["ブラウザ画面"]
        HTMX --> |2. 非同期展開| Browser
    end
~~~

### URL パラメータの責務分離

| Parameter | Role（役割） | Scope（影響範囲） |
|:---|:---|:---|
| `area` | 検索条件 | DB クエリ（結果一覧）の絞り込みに使用 |
| `parent_id` | UI 状態 | ドリルダウンのどの階層を展開するかを制御 |

---

## 3. 設計思想：URL as Single Source of Truth

すべての状態を URL に持たせることで、フロントエンドの複雑な状態管理を排除しました。

- **リロード＝状態復元**：ページをリロードしても、URL から「現在の検索結果」と「開いている階層」が自動的に再現される
- **ステートレスな遷移**：各ボタンは `location.href` による画面遷移をトリガーとし、ブラウザの「戻る」ボタンも標準機能として自然に動作する

---

## 4. サーバー駆動による UI 状態の復元（HTMX）

サーバー側で URL パラメータ（`parent_id`）を評価し、HTML の属性を動的に出し分けることで、フロントエンドに状態管理ロジックを持たせない（**No Client-side State**）実装を実現しています。

### 実装のメカニズム

1. **サーバーサイド判定**：Hono（SSR）が `parent_id` の有無を確認
2. **宣言的トリガー**：ID が存在する場合のみ、HTML に `hx-trigger="load"` 属性を付与して返却
3. **自動非同期ロード**：ブラウザが HTML を受け取った瞬間、HTMX が詳細階層を自動的に取得し、UI を展開

### メリット

- フロントエンドで URL をパースして命令的に処理する JavaScript を書く必要がない
- SSR 時に詳細なエリアリストを HTML に埋め込まずに済むため、初期レスポンスが極めて軽量

---

## 5. Path Enumeration による階層管理

階層データの管理には **Path Enumeration（経路列挙モデル）** を採用しました。

### 階層構造の例

~~~
10          （関東）
10-13       （東京都）
10-13-A001  （新宿区）
~~~

### 採用理由とメリット

- **D1（SQLite）との相性**：Recursive CTE（再帰クエリ）を使わず、`LIKE '10-13-%'` などの前方一致検索のみで高速に子階層を取得可能
- **親 ID 算出の軽量化**：文字列操作（スライス）だけで親階層を算出できるため、API ロジックを極めて単純化できる
- **ステートレス API**：パラメータ一つで「現在の位置」と「その直下の子要素」を一意に特定可能

---

## 6. UX（ユーザー体験）上の効果

本アーキテクチャにより、**「確定」と「さらなる絞り込み」の同時進行**が可能になります。

1. **アクション**：ユーザーが「関東」を選択
2. **同期的な結果更新**：画面全体がリロードされ、検索結果が「関東全体」で即座に描画される（SSR）
3. **非同期な UI 展開**：検索結果が表示されるのとほぼ同時に、ドリルダウン UI が「東京都・神奈川県…」という次の選択肢に自動展開される（HTMX）

「決定」ボタンを押す手間なく、クリックするたびに検索結果が絞り込まれていく、サクサクとした操作感を実現します。

---

## 7. 結論

本構成は、**Cloudflare D1 / Hono / HTMX** という軽量スタックを活かし、以下の3要素を高い次元で融合させています。

- **SSR** による高速描画・SEO・堅牢性
- **URL ベース**の状態管理による再現性と共有性
- **HTMX** による宣言的で軽量な非同期 UI

「検索エンジンとしての高い信頼性」と「モダンなアプリケーションの操作感」を最小限のコード量で両立する設計パターンであり、小〜中規模 Web サービスにおける最適解の一つと言えます。
