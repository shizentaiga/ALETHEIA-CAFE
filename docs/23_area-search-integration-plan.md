# エリア検索（市区町村拡張）＆ 現在地連携：実装計画書 (v2.1)

## 1. 設計思想：階層と自由度の両立
- **「全部」と「一部」の共存**: 各階層（都道府県・市区町村）のトップに「〇〇すべて」という選択肢を設け、広域検索を維持。
- **座標ベースの解像度向上**: 現在地取得時は、都道府県ではなく「市区町村」の代表点と照合し、より精度の高いエリア特定を行う。
- **スキーマの正規化**: `areas` テーブルを自己参照構造、または parent_id 形式に拡張し、ドリルダウンの動的生成を容易にする。

---

## 2. ドリルダウンのUI/UX再設計

ユーザーが迷わないよう、ドリルダウンを以下の3ステップ（または2ステップの動的変化）で構成する。

| ステップ | 選択内容 | 検索範囲 |
| :--- | :--- | :--- |
| **1. 地方** | 関東, 近畿... | 地方全体 |
| **2. 都道府県** | 東京都, 神奈川県... | 都道府県全体（ここで確定も可能） |
| **3. 市区町村** | **[東京都すべて]**, 千代田区, 江戸川区... | 選択した市区町村 |

- **「すべて」の導入**: 市区町村リストの先頭に「[都道府県名] すべて」を配置。
- **URL反映**: 市区町村選択時は `?area=13123`（江戸川区ID）、都道府県のみなら `?area=13`（東京ID）のように、IDの桁数やマスタ構造でスコープを判定。

## 4. 実装ロードマップ

### Step 1: 階層化座標マスタの構築（Data Layer）
1. **マスタ拡張**: `areas` テーブルを「都道府県」と「市区町村」が共存できる構造に変更。
2. **座標登録**: 各市区町村の代表点座標を登録。都道府県レコードには、その中心付近の座標を登録。

### Step 2: 階層型ドリルダウンの実装（API & View）
1. **`areaHandler.ts`**: HTMXを利用し、都道府県が選ばれたら即座に市区町村リストを返却・置換。
2. **UIの実装**: 「すべて」という選択肢が不自然にならないようデザイン（`headerStyle.ts`）を調整。

### Step 3: 現在地計算の精緻化（Shared Logic）
1. **最寄判定**: `navigator.geolocation` で得た座標から、全市区町村の中で最も近い `area_id` を特定。
2. **反映**: 現在地が「江戸川区」付近なら、ドリルダウンを自動で「関東 > 東京都 > 江戸川区」の状態に遷移させ、URLを `?area=auto:13123` とする。

## 5. 具体的実装のヒント

### ドリルダウンの「すべて」の扱い
市区町村の選択肢を生成する際、常に親要素（都道府県）を参照する特殊なレコードを先頭に挿入します。
~~~typescript
// イメージ
const options = [
  { id: parentId, name: `${prefName}すべて` }, // 都道府県全体
  ...cityList // 個別の市区町村
];
~~~

### 検索クエリの挙動（searchQuery.ts）
~~~sql
-- area_id が都道府県(例: 13)なら WHERE address LIKE '東京都%'
-- area_id が市区町村(例: 13123)なら WHERE area_id = 13123
~~~
※マスタ上で「都道府県ID」と「市区町村ID」を明確に区分（または親子関係で解決）することで、1つのパラメータで柔軟に検索範囲を変えられます。

---

## 6. 懸念事項と対策

- **リストの肥大化**: 市区町村数が多い（例：北海道）場合、ドリルダウンが長くなりすぎる。→ HTMXによる動的読み込みと、CSSによるスクロール領域の制限（`max-height`）で対応。
- **現在地精度の納得感**: 都道府県単位よりも市区町村単位で表示される方が、ユーザーは「自分の場所が正しく反映された」と感じやすいため、代表点マッチングは市区町村レベルで行う。
