## 優先度 A：ユーザー体験の向上とアプリ化（最優先）

「使い勝手」をネイティブアプリのレベルまで引き上げ、日常的に使われるツールにするフェーズ。

| 機能名 | 内容・メリット | 難易度 |
|:---|:---|:---:|
| PWA 対応 | ホーム画面への追加、アプリのような起動。再訪のハードルを最小化 | 低 |
| キーワード専用履歴 | ローカルストレージ（5件）保存。再検索を1タップで実行 | 低 |
| (⭐️実装完了)現在地取得機能 | ボタン一つで現在地を反映。PWA 化で「今ここ」の検索がより便利に | 低 |
| 最寄駅表示機能 | ショップ一覧に「〇〇駅から徒歩5分」等の情報を追加 | 中 |
| ショップ詳細ページ | 最小限のパラメータからページを動的生成。SEO 流入の受け皿 | 中 |

## コメント

全体の設計思想（A→B→C の順序）は筋が良い。ユーザーが定着してからマネタイズする構造は、プラットフォームビジネスの定石に沿っている。いくつか補足する。

### A の「低難易度3点」は同時に進めやすい

PWA 対応・キーワード履歴・現在地取得はいずれも独立して実装でき、互いに干渉しない。`manifest.json` と `service worker` の追加（PWA）、`localStorage` への5件保存（履歴）、`navigator.geolocation`（現在地）は、それぞれ1日以内で完成する規模感。まとめて一気にやるのが効率的。

### ショップ詳細ページは SEO 上の優先度が高い

難易度「中」に分類されているが、SEO 流入の受け皿として機能し始めると、他の機能の効果が加速する。Google ログインやお気に入り（優先度 B）は「詳細ページがある前提」で映えるため、A の後半で早めに着手する価値がある。

### 優先度 B の「ユーザー編集提案」はスパム対策を先に決める

コミュニティ編集はデータ品質とスパム投稿のトレードオフが生じる。「提案→承認フロー」か「即時反映＋通報」かの運用方針を、実装前に決めておくと管理コストが大きく変わる。

### 優先度 C の予約機能は Aletheia Core との統合を前提に

Aletheia Core（shizentaiga-2026）で構築中の予約・決済基盤がそのままこのフェーズに接続できる。優先度 C に達した時点で新規実装ではなく「既存基盤の統合」として扱えるため、他の機能より実質的な難易度は下がる可能性がある。
