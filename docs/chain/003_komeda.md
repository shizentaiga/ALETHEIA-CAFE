# コメダ店舗情報取得 API 仕様分析資料

## 1. エンドポイント基本情報
* **Base URL**: https://eu.komeda.co.jp/v1/hp/shop
* **Method**: GET
* **Content-Type**: application/json

---

## 2. クエリパラメータ (Request Parameters)
APIリクエスト時に指定可能なパラメータの構造です。

| パラメータ名 | 型 | 必須 | 内容・指定例 |
| :--- | :--- | :--- | :--- |
| prefecture | string | 任意 | 都道府県名。例: %E6%9D%B1%E4%BA%AC%E9%83%BD (東京都) |
| brand_type | number | 任意 | ブランド種別。通常は 1 (コメダ珈琲店) を指定。 |
| offset | number | 重要 | 取得開始位置。1ページ20件ずつのため、0, 20, 40...と加算してループ回す。 |

---

## 3. レスポンスデータ構造 (Response JSON)

### A. 全体構成
(JSON構造)
{
  "total": 102,
  "fields": [...],
  "items": [...]
}

### B. items (店舗詳細配列)
プログラムで抽出・保存すべき主要プロパティです。
* **id** (number): システム固有の店舗ID
* **name** (string): 店舗名（例: "町田鶴川店"）
* **address** (string): 所在地住所
* **brand_type** (number): ブランドID

### C. fields (マスタ情報)
店舗詳細には含まれない「設備名」や「サービス名」のIDと名称の対応表です。
* **207-1**: 禁煙／喫煙
* **207-7**: WiFi
* **207-8**: 電源

---

## 4. Playwright 実装用ロジック・アルゴリズム

### ステップ1：全件数の把握とループ
1. 初回リクエスト（offset=0）を投げ、total（例: 102）を取得する。
2. Math.ceil(total / 20) で総ページ数を算出する。

### ステップ2：ページング処理
* offset を20ずつ増やしながら、全ての items を配列に結合（push）していきます。

### ステップ3：実装時の注意点 (Tips)
* **User-Agent**: ブラウザからのアクセスを装うため、Playwrightのデフォルト設定、または適切なヘッダー設定を推奨します。
* **URLエンコード**: 都道府県名は encodeURIComponent('東京都') で処理してください。

---

## 5. データ抽出用 JSON Path
* **店舗名のリスト**: $.items[*].name
* **総件数**: $.total
* **ブランド名（マスター）**: $.fields[?(@.name=="brand_type")].values

# 調査報告：動的Webサイトからの店舗データ抽出手法について

## 1. 概要
対象ウェブサイト（コメダ珈琲店 店舗検索ページ）は、HTMLソース内に直接データを持たない動的コンテンツ（JavaScriptレンダリング型）である。本調査では、ブラウザの通信を解析し、バックエンドAPIから直接JSONデータを取得する手法を特定した。

---

## 2. 解析対象の特定
当初のHTMLソース（静的解析）では、データ部分が v-text 等のテンプレート属性になっており、情報の直接取得が不可能であった。

*   **HTML側の記述**: `<li v-text="$?.shop?.name"></li>`
*   **結論**: ブラウザ実行時に外部APIからデータをロードしていると判断。

---

## 3. 通信解析（ネットワーク・フォレンジック）の手順
ブラウザの開発者ツール（Networkタブ）を用い、以下の手順でデータソースを特定した。

1.  **フィルタリング**: 通信種類を `Fetch/XHR` に限定。
2.  **イベント発火**: 画面上の「検索」を実行し、発生した通信を捕捉。
3.  **エンドポイントの特定**: google-analytics 等の統計用通信を除外。
4.  **ヒットURL**: `https://eu.komeda.co.jp/v1/hp/shop`

---

## 4. 取得データの構造解析
特定したAPIから返却されるデータは、構造化された JSON形式 である。

| 項目名 | データ型 | 内容 |
| :--- | :--- | :--- |
| **total** | Number | 検索条件に合致する全店舗数 |
| **items** | Array | 各店舗の詳細情報のリスト |
| **items[].name** | String | 店舗名 |
| **items[].address** | String | 住所 |

---

## 5. 効率的な収集手法（今後の実装方針）
HTML要素をパース（スクレイピング）するのではなく、APIリクエストを模倣（APIコール）することで、高速かつ正確なデータ収集が可能となる。

*   **ページネーションの制御**: URLパラメータの `offset` 値を20刻み（0, 20, 40...）で変更することで、全件を網羅。
*   **自動化ツール**: Playwright等の `waitForResponse` メソッドを使用して、通信レスポンスを直接オブジェクトとして処理する。
