# 03_データベース移行計画書（Database Migration Plan）

## ファイル名: docs/03_migration-plan.md

## 1. 概要
本プロジェクトのエリア管理を、`constants.ts` および `services` テーブルの文字列管理から、`areas` テーブルによる階層 ID（`area_id`）管理へ移行するための計画である。本移行は、既存の検索機能（都道府県検索）を維持したまま、段階的に実施する。

---

## 2. 移行フェーズ

### ステップ 1：エリアマスタの構築と基盤整備
1. **エリアテーブルの追加**:
   - `areas` テーブルを `schema.sql` に追加し、デプロイする。
   - `area_level` (1:地方, 2:都道府県, 3:市区町村/仮想) を含める。
2. **シードデータの準備 (HeartRails API 活用)**:
   - HeartRails API より日本の全市区町村データを取得し、`area_id`（JISコードベース + ゼロ埋め）を生成する。
   - 北海道等の特定地域に対し、仮想エリア（道央・道東等）を定義し、親子関係を `area_id` の前方一致ルールで構成する。
3. **影響調査**:
   - テーブル追加のみで、既存の `services` テーブルへのクエリが破壊されないことを確認し、コミットする。

### ステップ 2：既存店舗データの `area_id` 紐付け
1. **services テーブルの拡張**:
   - `services` テーブルに `area_id` カラムを追加（NULL 許容で開始）。
2. **既存シードデータの修正 (スタバ・ドトール・小岩店舗等)**:
   - `pref` と `city` の文字列を元に、対応する新 `area_id` を特定し、シードデータを書き換える。
   - **移行期間の共存**: この時点では `pref`, `city`, `area_id` の 3 カラムが共存する。
3. **動作確認**:
   - 既存の文字列ベースの検索と、新しい `area_id` ベースの検索が同等の結果を返すことを確認する。

### ステップ 3：完全移行とクリーンアップ
1. **ソースコードの置換**:
   - `constants.ts` からの読み取りを廃止し、`areaHandler.ts` 経由の DB 取得へ切り替える。
   - 全ての検索ロジックを `WHERE area_id = ?` または `LIKE ?` へ書き換える。
2. **最終クリーンアップ**:
   - `schema.sql` から `pref`, `city` カラムを削除し、`area_id` を `NOT NULL` 制約に変更する。
   - シードデータから旧カラムの定義を完全に削除し、新フォーマットのみに統一する。

## 3. 懸念事項と対策（考慮漏れチェック）

### 3.1 住所変更・市町村合併への対応
- **懸念**: HeartRails API で取得したデータと、既存店舗の `city` 名が微細に異なる（例：ケバ取り、表記揺れ）場合にマッピングが漏れる。
- **対策**: ステップ 2 の移行時に、マッピングに失敗したレコードを抽出するチェッククエリを実行し、手動または正規化ロジックで補正する。

### 3.2 area_id のリナンバリング
- **懸念**: 運用開始後に「仮想エリア」の区切りを変更したくなった場合、`services` テーブルの大量のレコードを更新する必要がある。
- **対策**: DB 移行計画書に記述した通り、物理的な外部キー（FOREIGN KEY）は張らずに「論理紐付け」とすることで、一括 UPDATE の柔軟性を確保する。

### 3.3 パフォーマンスの低下
- **懸念**: `LIKE '13-%'` などの前方一致検索が増えることで、D1 のインデックス効率が懸念される。
- **対策**: `area_id` に INDEX を張ることは必須とし、件数が増えた場合は `LENGTH(area_id)` を組み合わせたインデックスの検討も行う。
