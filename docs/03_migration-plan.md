# 03_データベース移行計画書（Database Migration Plan）

## ファイル名: docs/03_migration-plan.md

## 1. 概要
本プロジェクトのエリア管理を、`constants.ts` および `services` テーブルの文字列管理から、`areas` テーブルによる階層 ID（`area_id`）管理へ移行するための計画である。既存機能を維持しつつ、データの整合性を担保した段階的な移行を目指す。

---

## 2. 移行フェーズ

### ステップ 1：エリアマスタの構築と基盤整備 【完了】
1. **エリアテーブルの追加**:
    - `areas` テーブルをデプロイ。`area_id` を前方一致検索（LIKE）に最適化した設計で実装済み。
2. **シードデータの準備**:
    - 全市区町村データを取得し、階層構造（1:大, 2:中, 3:小）を持つ ID を生成・投入済み。
3. **影響調査**:
    - テーブル追加後も既存動作に影響がないことを確認し、ベースラインを確立。

### ステップ 2：既存店舗データの ID 紐付け 【完了】
1. **services テーブルの拡張**:
    - `services` テーブルに `area_id` カラムを実装。
2. **既存シードデータの修正と ID 付与**:
    - `starbucks.sql`, `doutor.sql`, `koiwa.sql` 等の主要シードを新フォーマットへ移行。
    - **サブクエリによる動的紐付け**: `INSERT` 時に `areas` テーブルから `area_id` を自動取得する仕組みを導入。
3. **動作確認**:
    - ローカルおよびリモート（D1）環境への適用を完了。エリア検索が正常に動作することを確認。

### ステップ 3：完全移行とクリーンアップ 【進行中】
1. **ソースコードの整理（進行中）**:
    - `constants.ts` 内の静的な地理情報配列を廃止・整理し、DB（`areas` テーブル）参照へ一本化。
2. **ロジックの完全置換**:
    - エリア解決（`resolveDetectionArea`）やドリルダウン機能を `area_id` 前方一致に統一。
3. **DB クリーンアップ（未着手）**:
    - 移行期間用の旧カラム（`pref`, `city`）を物理削除し、スキーマを最適化。
    - **運用**: アプリケーション層で `area_id` の必須チェックを保証。

---

## 3. 実装上の重要ポイント（考慮漏れ対策）

### 3.1 フォールバック検索の維持
`pref`/`city` カラム削除後も、`address`（フル住所）に対する部分一致検索を維持することで、マスタ未登録エリアやキーワード検索時のユーザー体験を担保する。

### 3.2 パフォーマンスとインデックス
D1（SQLite）において `TEXT` 型の `LIKE '13-%'` 検索はインデックスのレンジスキャンが効くため、以下のインデックスが定義されていることを維持する。
~~~sql
CREATE INDEX idx_services_area_id ON services(area_id);
~~~

### 3.3 データ整合性のクロスチェック
ステップ3の完了前に、以下の SQL で紐付け漏れがないかを最終検証する。
~~~sql
-- area_id が付与されていない店舗の抽出
SELECT service_id, title, address FROM services WHERE area_id IS NULL;

-- area_id がマスタに存在しない店舗の抽出
SELECT s.service_id FROM services s 
LEFT JOIN areas a ON s.area_id = a.area_id 
WHERE a.area_id IS NULL;
~~~
